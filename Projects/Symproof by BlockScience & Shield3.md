## Description

# **Symproof: Uncovering Hidden Axioms in Ethereum Protocol Design by Formalizing the "Math Layer" of Ethereum Security**

  


A project providing essential Infrastructure for institutional-grade security and identifying "unknown unknowns" before they become multi-million dollar exploits, Symproof is an extension of BlockScience’s longstanding work in creating, distributing, and stewarding Ethereum security public goods. The Symproof repo is a modular extension, working prototype, and functional automation of the existing, widely-adopted [cadCAD](https://cadcad.org/), [MSML](https://github.com/BlockScience/MSML), and Token Engineering tools, frameworks, and practices we have stewarded since 2017. It codifies both our longstanding open-source and proprietary work in the underlying math and security logic that has protected billions in TVL and refined grant programs by flagging and reducing fraud. It enables the automation and scaling of this security work as a public good and the formal proof-checking extension for the cadCAD modeling ecosystem.

  


# **Project Overview**

  


Ethereum security currently operates at primarily two layers: manual auditing and formal code verification. [Symproof](https://github.com/BlockScience/sympy-proof/tree/main) is an open-source library built on [SymPy](https://www.sympy.org/en/index.html) that adds a third layer - verifying that the mathematical formulas underpinning a protocol have their claimed properties before any code is written. 

  


Tools like [Certora](https://www.certora.com/) verify code correctness, while tools like [Slither](https://github.com/crytic/slither) find common vulnerability patterns in code. Neither is designed to verify that the specification's math is correct in the first place. A class of security failures involves mathematical invariant violations that were present at the specification level - conditions that downstream tools inherit as assumptions rather than check.

  


Symproof allows researchers to write small, focused, hashed proofs about formulas. Each proof answers one question: does this formula have this property, given these assumptions? It uses SymPy's simplify(), ask(), and refine() for symbolic verification, and srepr() for deterministic serialization of expressions into proof hashes. Assumptions are explicit and hashed into the proof identity - changing an assumption changes the hash. Inherited axioms (conditions forced by an external theorem rather than chosen by the proof author) must carry a Citation (source="..."). The full chain from a protocol's proof back to the academic result it depends on is traceable by design. This allows for surfacing, documenting and testing the mathematical underpinnings of DeFi, ZK, and other protocols. 

  


As Symproof is built on SymPy, auditors are one step from numerical evaluation. SymPy expressions can be passed directly to [lambdify()](https://docs.sympy.org/latest/modules/utilities/lambdify.html) to produce numerical Python functions and test vectors, moving from a symbolic proof to concrete validation without leaving the tool. Where SymPy's verification is heuristic rather than complete, the proof says so. The result carries an advisory flagging the specific concern and the step it applies to.

  


Proof bundles are permanent and composable. Instead of mathematical facts about a formula pattern being re-derived independently for each audit, a verified bundle can be published and imported by hash into future work. This makes security research cumulative across engagements rather than within them and allows for further Public Goods development across protocol security domains. Future projects could import these bundles (e.g., "Uniswap v3 Tick Arithmetic Invariants") rather than re-deriving them, raising the security floor for the entire ecosystem.

  


# **The Gap & Our Proposal**

  


## **The Problem: The "Hidden Axiom" Gap**

  


In practice, teams often cite mathematical theorems without verifying whether Ethereum's operational constraints satisfy the theorem's implicit conditions. The knowledge produced by each audit then lives in static PDFs, where every auditor re-derives the same facts independently, and errors propagate through every downstream layer until they surface as exploits. No practical tool exists for making formula-level claims machine-checkable and reusable before code is written.

  


High-reliability systems fail when their environment violates their mathematical assumptions. If an engineer cites a theorem without checking its preconditions against Ethereum's constraints, the proof is incomplete, while the gap may not be visible until it is expensive.

  


## **The Solution:** **Machine-Checkable, Cumulative Security**

  


Symproof makes mathematical assumptions explicit, hashed, and independently verifiable. It introduces three properties to the security workflow that do not currently exist together in any available tool:

  


  * Assumption Transparency. All axioms are explicit and hashed. If an assumption changes, the proof hash changes. Hidden simplifications become hard errors, as the framework refuses to proceed if a proof depends on a condition that isn't declared.
  * Decomposition. Complex protocols are broken into focused Proof Bundles. Rounding Direction, Overflow Bounds, Entropy Leakage - each reviewable independently. Specialists can verify one property without reading the rest.
  * Honest Confidence. When verification reaches a known limitation of the underlying symbolic engine, Symproof attaches an Advisory rather than passing silently - explicitly flagging where human review is still required. Beyond advisories, seal() actively rejects provably false axioms, contradictory axiom pairs, and proofs where lemma assumptions contradict the declared axiom set. The framework is designed to fail loudly rather than pass silently.



  


Together these properties address a structural problem in Ethereum security: research is not always cumulative. By automating the derivation of mathematical ground truth, Symproof allows human auditors to focus on genuinely novel risks, reducing the cost of thorough security review across the ecosystem and creating a multiplier effect on auditor productivity. This is especially valuable as the complexity of deployed protocols continues to outpace the supply of qualified reviewers. Symproof additionally ships with structured LLM guidance (as a Claude Code [skill.md](http://skill.md/) file) that lowers the barrier to writing correct proofs, making the framework accessible to security engineers who are not specialists in symbolic mathematics.

  


# **What This Is vs. What It Is Not**

  


## **What Symproof Is**

  * A Pythonic framework built on SymPy for making the mathematical assumptions of DeFi and ZK protocols explicit, hashed, and independently verifiable
  * A design-phase tool - it operates before Solidity is written, on the formulas themselves
  * A growing library of reusable, pre-proved components for common Ethereum formula patterns: AMM arithmetic, ZK circuits, fixed-point rounding, vault invariants
  * Accessible to security engineers who are not specialists in symbolic mathematics - the repo ships with structured LLM guidance that assists in writing correct proofs
  * Equipped with falsity and consistency guards - seal() rejects self-contradictory axiom sets and proofs that depend on undeclared load-bearing assumptions, before verification runs



  


## **What Symproof Is Not**

  * A replacement for Certora, Halmos, or Slither - it sits beneath them, ensuring the specification those tools verify is itself sound
  * A Solidity verifier or runtime monitor
  * A full theorem prover - it is less rigorous than Lean or Coq, more rigorous than auditor inspection, and designed for the workflow of practicing security engineers



  


In short: Symproof tells you what properties to prove and verifies the math behind them - formal verification then checks that the code delivers those properties, and fuzzing finds edge cases at runtime. Each layer depends on the one below it being sound.

  


# **Applications**

  


## **AMM Mathematical Audits**

Symproof allows auditors to verify AMM formula properties at the design stage: swap output positivity under declared reserve and fee conditions, rounding direction at each step of an execution pipeline, uint256 overflow safety under realistic reserve sizes, and correct cross-decimal normalization (e.g., USDC at 6 decimals against WETH at 18). Each property is a separate, independently verifiable proof bundle. As such, a finding on overflow safety does not depend on the rounding proof being read first.

  


## **ZK Circuit Security & Soundness**

Symproof supports verification of R1CS witness satisfaction and symbolic analysis of output entropy. An AND gate provably leaks more information about its inputs than an XOR gate - symproof can quantify this difference as a Shannon entropy bound. This gives researchers a symbolic handle on information leakage that complements but does not replace cryptographic soundness proofs.

  


## **Protocol Specification & Hidden Axioms**

When a protocol cites an external theorem to justify a security property, it inherits that theorem's preconditions. If those preconditions are not explicitly declared, the proof has a gap. The theorem holds in the abstract but may not hold under the protocol's specific constraints. Symproof surfaces these hidden axioms by requiring every assumption to be declared and hashed into the proof. A proof that depends on an undeclared condition is a hard error, not a warning. Where an axiom is inherited from an external theorem, it must carry a Citation naming the source. The assumption chain is traceable from the protocol's proof bundle back to the named academic result. Not just declared, but attributed.

  


## **Rounding Pipeline Integrity**

Multi-step DeFi pipelines accumulate rounding errors across deposit, swap, fee, and withdrawal operations. The direction of each rounding (whether it favors the pool or the user) determines whether the protocol is sound or leaking value. Symproof proves the net rounding direction across an entire pipeline symbolically, identifying directional errors before the code is written. Rounding bugs in this class have been exploited in production; catching them at the formula level is substantially cheaper than catching them after deployment.

  


# **Working Output & Further Technical Details**

  


The project is not just a proposal, but already has a functional, installable, Apache 2.0 licensed codebase with a test suite, domain libraries, and runnable examples.

**Repository:** BlockScience/sympy-proof

**License:** Apache 2.0 (currently MIT, see Issue #10) 

**Dependencies:** SymPy ≥ 1.12, Pydantic ≥ 2.0, Python ≥ 3.12

  


The library in its current form includes:

  * A proof engine (seal(), verify_proof(), verify_lemma()) that enforces assumption transparency, hash binding, and honest confidence via the advisory system
  * Domain libraries covering AMM arithmetic, ZK circuits, fixed-point rounding, convex optimization, linear programming, control systems, and Shannon information theory
  * Runnable audit examples: including a full AMM swap audit producing eight independent proof bundles and a DIP routing case study that surfaced seven hidden axioms across three proof pairs
  * A ProofBuilder chainable API for constructing proof scripts without boilerplate
  * Structured LLM guidance for assisted proof writing, lowering the barrier for security engineers who are not specialists in symbolic mathematics



All proofs are deterministic: running the same proof twice produces the same SHA-256 hash. The codebase is managed with uv and linted with ruff.

  


# **Roadmap & Further Work: From Core Engine to Ecosystem Standard**

  


We consider the Symproof core engine as functional and the foundational design decisions as proven, but a working research tool and a usable public library are different things. The roadmap describes the sequence of work required to close that gap, and to build the domain coverage that makes the tool valuable beyond the team that built it. The three phases follow a deliberate dependency order.

  


**Phase 1** is a prerequisite for everything else. An open-source library that external contributors cannot install, understand, or extend reliably is not a public good. The provenance infrastructure (citation tracking, inherited axiom marking, foundation enforcement) is stabilized here as the documented backbone that all domain libraries build on.

  


**Phase 2** establishes credibility and accessibility through application. Domain libraries built on an unvalidated tool are unconvincing. A public, reproducible analysis of a nine-figure incident (the Balancer V2 rounding exploit) demonstrates what the tool can actually catch, where its scope ends, and how it connects to the broader verification stack. This is the reference case that makes Phase 3 libraries trustworthy rather than theoretical.

  


**Phase 3** is where the compounding effect begins. Each domain library published as a hashable proof bundle is a permanent contribution to the ecosystem. An auditor reviewing a Uniswap v3 fork two years from now can import a verified tick arithmetic bundle rather than re-deriving the mathematics. The value accumulates across every subsequent audit that uses the same formula patterns, where audit by audit, the baseline rises.

  


The extent of Phase 3 scales with available funding. The priority order is listed in the Use of Funds section. The measure of success is not components shipped but whether a practicing Ethereum security engineer can reduce their mathematical re-derivation work drastically for the covered patterns.

## **Tier 2: Case Study ($25k–$50k)**

_Focus: Demonstrating the tool against a real exploit with documented impact (extent pending actual funding amount reached)_

_Context:_ The Balancer V2 Composable Stable Pool exploit (November 2025, $94.8M) was not a code bug. The Solidity code executed correctly. The flaw was in the mathematical specification: the <__upscale >_ function rounded <_amountOut >_ down through the exact-out path, causing <_amountIn >_ to also round down, meaning callers systematically underpaid and the pool invariant decreased with every swap.

  * This is a formula rounding direction property. It is stateable as a symproof hypothesis and the directional chain proof through <__upscale_ → StableMath> would have failed pre-deployment, surfacing the gap before any code was written.



  


The deliverable is a fully public, worked Symproof analysis of the exploit:

  * Domain library correctness audit prior to case study publication: convex_hessian() (Issue #1) and lyapunov_stability() (Issue #2) fixes, plus a systematic review of all domain library functions for similar criterion errors
  * A complete proof bundle targeting the rounding direction property of the exact-out swap path
  * Explicit hidden axiom enumeration: what conditions the stable swap math implicitly requires that the V2 specification never declared
  * A documented finding in symproof's evidence format: hypothesis, axiom set, proof failure, and the corrected hypothesis that V3's architecture satisfies
  *   




The report will also sketch how simulation and formal verification would extend the finding: characterizing exploitability under different liquidity conditions, and asserting the corrected invariant in code. The case study will trace the full verification chain from symbolic proof of the formula flaw, through simulation of exploitability conditions, to formal assertion of the corrected invariant in code, demonstrating where symproof fits in that chain and what each subsequent layer adds, even where those layers are illustrated rather than fully executed.

  


_Scope:_ symproof surfaces the formula flaw. Connecting it to the full attack path requires human judgment and additional tooling. The case study documents that boundary explicitly.

_Outcome:_ a public, reproducible analysis of a nine-figure incident demonstrating exactly what formula-level verification catches and where it hands off to the next layer.

  


## **Tier 3: Domain Libraries (e.g.: DeFi, ZK, …) (+$50k)**

_Focus: Reusable proof components for the most common Ethereum formula patterns (extent pending actual funding amount reached)_

Builds on Tier 1 infrastructure and Tier 2 Case Study. All work is greenfield mathematical formalization e.g.:

**Uniswap v3 concentrated liquidity**

  * Tick boundary arithmetic: 1.0001^tick encoding, square root price representation, boundary conditions at tick transitions
  * Token amount formulas: provable bounds on token quantities as a function of liquidity and price range
  * Price impact bounds: symbolic upper bounds on output shortfall as a function of input size and in-range liquidity depth



**Generalized AMM invariants**

  * Stable swap curves: invariant conservation under swap, rounding direction through the Newton iteration, output positivity under the Curve formula
  * Weighted pools: power function invariant conservation, rounding direction, output bounds as a function of weight parameters



**Fixed-point arithmetic**

  * Directed rounding proofs for WAD and RAY scale operations
  * Chained error bounds across multi-step pipelines
  * Overflow safety conditions parameterized by reserve size



**Sandwich attack profitability bounds**

  * Symbolic upper bounds on sandwich profit as a function of pool depth and input size — formula-level only
  * Does not model mempool ordering, gas competition, or partial fills



**ZK circuit expansion**

  * R1CS witness satisfaction for multi-constraint systems
  * Output entropy bounds for single and small composed gate circuits (larger circuits likely require research beyond SymPy's current capabilities)
