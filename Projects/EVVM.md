## Description

### Have you ever thought of Ethereum as a hypervisor for virtual blockchains?

###   


The next layer of Ethereum security and scalability isn't another chain. It's a substrate where many independent virtual blockchains run as smart contracts on top of the EVM, each with its own rules for accounts, gas, tokens, permissions, signatures, and compliance. No new infrastructure, no sequencers, no bridges, no liquidity fragmentation. **That's  EVVM.**

  


###   


### Security work

Most security tooling on Ethereum operates _on top of_  the chain: audits, monitors, incident response, governance security. EVVM extends the design space _inside_  it. Gates, signature schemes, compliance rules, privacy layers, and post-quantum research can be deployed and iterated as virtual blockchains, without disrupting the host chain.

  


What's missing on plain EVM today, and where EVVM fills the gap:

  * Chain compliance: KYC/AML and access control as native onchain primitives.
  * Token incompatibilities: e.g. x402 payment-required flows for the agent economy and new ones.
  * Complex privacy solutions: zk + FHE, composable with the public chain.
  * Post-quantum resistance: signature scheme migration without losing identity, at account level.
  * Gas and account abstraction: without external alternative-mempool stacks/new infra.
  * Multi-chain composability: virtual blockchains can be deployed across host chains without each becoming an isolated environment.



  


  


### How it works

EVVM is as decentralized as each deployer wants and as closed as each purpose needs. Core contracts and admin functions are renounceable, so a deployment can be fully immutable. Fishers (relayers) are optional helpers that improve UX, run markets, and can be configured to emulate sequencers or sharded execution; users can also be their own relayers, if permitted by the deployer. This makes EVVM a substrate that adapts to use cases ranging from open DeFi to permissioned bank consortiums, all on the same protocol.

  


Signatures today use the EIP-191 standard. The architecture is built so signature schemes evolve cleanly, that's what makes the post-quantum migration tractable (see "What's coming" below).

  


### The last changes

  1. Cryptographic primitive abstraction in [Core](https://github.com/EVVM-org/testnet-Contracts/blob/main/src/contracts/core/Core.sol): A single, centralized validation entry point in Core verifies signatures, nonces, and executor permissions atomically. This makes it possible to evolve signature schemes (passkeys, contract-account signatures, post-quantum) without rewriting downstream services. Signature abstraction is the load-bearing primitive for the next wallet UX and post-quantum migration.
  2. zkVVM: privacy for users, transparency for institutions. A zero-knowledge service for EVVM for private token transfers on Ethereum. Designed so fund movements among institutions remain auditable while transactions among compliant users can be 100% private (via zkProofs). [MVP ready to deploy by anyone](https://github.com/EVVM-org/zk_vvm), new modular release on April 30th.
  3. KYC/AML/Whitelist gates as native primitives: Compliance is a security concern when the alternative is opaque mixers and unmonitored flows. EVVM lets each virtual blockchain define its own access control using KYC/AML gates, allowlists, token-gated access, custom delegatecall checks. Banks and institutions can experiment with compliance-aware environments without operating their own chain.
  4. x402: programmable payment-required middleware. TypeScript middleware for Express.js / Next.js that accepts EVVM payments via the HTTP 402 status code. [Published on npm](https://github.com/EVVM-org/x402) as @evvm/x402 v0.1.6. Enables AI agents, APIs, and machine-to-machine flows to pay-per-call with cryptographically verified payments, an important security primitive for the agent economy.
  5. Sandbox infrastructure for safer development: [scaffold-evvm](https://github.com/EVVM-org/scaffold-evvm) provides a complete dev environment with 30-second timelocks instead of production's 30-day timelocks, letting security researchers and developers iterate quickly on EVVM-based security designs locally before deploying. Now ships with a built-in block explorer, making it one of the most complete sandbox environments in the EVM tooling space.

### What's coming next: QR-EVVM (July 2026 target)

A working EVVM where accounts use post-quantum signature schemes instead of EIP-191. The novel part: users keep stable on-chain identity through usernames (already resolved by our NameService contracts today), so when key formats and signature algorithms migrate, users don't lose who they are.

  


Three primitives line up to make this practical:

  * Usernames as fixed IDs: already shipped in NameService.
  * Gas abstraction: already shipped on the current EVVM; in QR-EVVM it covers post-quantum signatures so the user flow stays gasless and identical.
  * Composability with the host chain: via abstraction services so QR accounts interact seamlessly with the rest of the EVM ecosystem, e.g. Uniswap, AAVE, Stablecoinns, Payments, RWAs, etc.



  


Adjacent work also on the active roadmap:

  * FHE-EVVM for confidential accounting on Ethereum.
  * Hyperledger Besu EVVM consortium for banks (treasury on Ethereum, public products listing), 30 minutes presentation made at Decentralized Trust Linux Foundation event, in exploration with NTT Data.
  * Abstractor Service and Storage Service as new on-chain services in the next-month roadmap.

### Why this round for EVVM?

The Ethereum Security QF round explicitly funds: security tooling & infrastructure, cryptographic primitives, ZK & circuit security, post-quantum research, secure development tooling. EVVM hits four of those at once and extends the Ethereum security space _inside_  the chain A platform where new compliance, privacy, and post-quantum primitives can be built, tested, and shipped without launching new chains.

  


This is also the first round of TheDAO Security Fund, the same fund whose reactivation depended on careful on-chain assertions and batched transactions. EVVM's primitives, typed signatures, async nonces, batched execution, central transaction validation, are exactly the building blocks the next generation of those careful operations will rely on.
